## 结论

**保持 `tts/` 模块原状，本阶段不拆**。

## 评估依据

蓝图原文（plan v2）："tts/ 模块去留评估（plan v2 误判后移到这里，本 Phase 评估，不一定拆）"。

## 与 god class refactor 范围的关系

- 最大单文件 `TtsController.kt` 311 行，仍远低于 app 主模块 god class 阈值（500+/1000+）
- 接口 + 实现分离已做（TTSProvider + 8 个 provider 文件）
- 控制器与 provider 解耦已做（controller/ 与 provider/）
- 没有上帝类 / 没有违反 SRP / 没有循环依赖
- 最大文件 TtsController 内部按 phase（synthesize / chunk / play / state）逻辑分段清晰，无 god method

## 命名一致性

蓝图原文提到的 "tts/ 命名" 关切，应该是指与 app 主模块的 `me.rerere.rikkahub.data.ai.tts.*`（如果存在）对齐。当前包名 `me.rerere.tts.*` 是独立 Gradle module 的标准命名，无需对齐 app 主包。

## 后续建议

如果未来 tts 增加复杂度（如多语言细分、流式合成新策略等），再评估拆 controller/ 内部子文件。当前不动。
