# 今日看板 — 产品功能方案

> 版本: v0.1  
> 日期: 2026-05-10  
> 状态: 方案设计阶段

---

## 一、功能定位

**Agent 主动产出的个人日更面板。**

通过多数据源采集 + 规则粗筛 + Agent 推理，生成结构化的"今天你应该关注什么、做什么"。不是被动的通知中心，而是一个能思考、能建议、能学习的智能助理面板。

### 核心价值

- **信息降噪**：从几十上百条通知/消息中，提炼出真正值得你关注的 5-10 条
- **主动建议**：不只告诉你"发生了什么"，还告诉你"你应该做什么"和"下一步建议"
- **陪伴感**：随着一天的进展持续更新，越用越懂你

### 与现有功能的关系

- 复用 Feishu Document Radar 的文档变更监控能力
- 复用现有 MCP 通道获取飞书消息/日历
- "聊一下"功能与现有 Chat 系统无缝衔接

## 二、用户故事

| 角色 | 场景 | 期望 |
|------|------|------|
| 用户 | 早上打开 app | 看到今天要做的事、值得关注的事，快速进入工作状态 |
| 用户 | 下午再看一眼 | 看板已更新，新的消息被整合进来，已完成的事项标灰 |
| 用户 | 看到一条建议想深入 | 点"聊一下"，Agent 已经知道背景，直接开聊 |
| 用户 | 某类通知总是不关心 | 连续忽略几次后，Agent 自动不再推这类内容 |
| 用户 | 想让 Agent 盯着某件事 | 在设置里添加关注点，Agent 后续会重点关注相关信号 |

## 四、数据源

### MVP 全部接入

| 数据源 | 采集方式 | 信号类型 | 依赖 |
|--------|---------|---------|------|
| 系统通知 | NotificationListenerService | 实时 | 需用户授权通知读取 |
| 日历 | CalendarProvider + ContentObserver | 变更触发 | READ_CALENDAR 权限 |
| 飞书消息 | MCP 轮询 / WebSocket | 准实时 | 需 MCP 配置 |
| 飞书文档变更 | 复用 DocuP 配置 |

- 邮件（IMAP/飞书邮箱）
- GitHub/GitLab 通知

## 五、触发策略

### 三层触发机制

| 层 | 机制 | 作用 | 技术实现 |
|---|------|------|---------|
| 实时采集 | NotificationListenerService | 系统通知实时捕获，写入 SignalStore | 系统级常驻服务，不受后台限制 |
| 定时推理 | WorkManager PeriodicWork | 定时检查信号量，触发 Agent | 最小间隔 15 分钟，系统保证执行 |
| 前台补偿 | App 启动/恢复前台 | 检查上次运行时间，超阈值立即跑 | 无限制 |

### 触发条件

- **定时节点**：默认早 8:00 / 午 12:00 / 晚 18:00（用户可配置）
- **增量阈值**：新信号累积 ≥ 5 条时触发更新（用户可配置）
- **手动触发**：用户下拉刷新时立即执行
- **前台补偿**：上次 Agent 运行超过 2 小时 + app 回到前台 → 自动触发

### 后台执行保障

```
NotificationListenerService（常驻）
    │
    │ 实时写入信号
    ▼
SignalStore (Room DB)
    │
    │ WorkManager 定时检查
    ▼
信号量 ≥ 阈值？──→ 触发 BoardAgent
    │
    │ 或定时节点到达
    ▼
触发 BoardAgent
```

**电量控制：**
- NotificationListenerService 本身极轻量（仅接收回调写 DB）
- Agent 推理一天最多 5-8 次（3 次定时 + 几次增量），每次几千 token
- 可配置"仅 WiFi 下执行"或"仅充电时执行"非紧急推理

## 六、Agent 推理设计

### 输入

每次触发时，打包以下内容给 Agent：

```
System Prompt:
- 角色定义：你是用户的个人助理，负责分析信息并生成今日看板
- 用户画像/偏好: {prompt_memory}
- 用户配置的关注点: {focus_rules}
- 当前时间: {now}
- 输出put_schema}

User Message:
- 今天收集到的全量信号: {signals_json}
```

### 输出 Schema

```json
{
  "summary": "今天有 3 个会议，2 件需要回复的事，1 个文档更新值得看看。age",
      "source_ref": "msg_id_xxx",
      "source_content": "原始信号的关键内容摘要",
      "urgency": "high",
      "category": "action",
      "reason": "张三昨天发的，提到周三前需要回复",
      "suggestion": "可以先确认数字再回复，需要我帮你拉一下上季度数据吗？",
      "ti26-05-10T09:15:00+08:00"
    }
  ]
}
```

**字段说明：**

| 字段 | 类型 | 说明 |
|------|------|------|
| id | string | 唯一标识，用于状态追踪 |
| title | string | 一句话描述，显示在看板上 |
| source_type | enum | 来源类型：notification / calendar / feishu_message / feishu_doc / chat_history / time |
| source_ref | string | 原始信号的引用 ID，用于溯源 |
| source_content | string | 原始信号的关键内容，用于"聊一下"时传递上下文 |
| urgency | enum | high / medium / low |
| category | enum | action（要做的事）/ attention（值得关注）/ info（今日动态） |
| reason | string | Agent 为什么觉得这条值得你关注 |
| suggestion | string | Agent 的下一步建议 |
| time | datetime | 信号发生时间 |

### 模型选择

- 默认跟随用户配置的主模型
- 可在设置中单独指定看板专用模型（如用更便宜的模型控制成本）

## 七、反馈学习机制

### Level 1：权重规则引擎

| 用户行为 | 系统反应 |
|---------|---------|
| ✓ 完成 TODO | 该来源/关键词权重 +1 |
| 💬 聊一下 | 该来源/关键词权重 +2（深度关注） |
| × 忽略 | 该来源/关键词权重 -1 |
| 连续 3 次忽略同一来源 | 自动降级，不再出现在看板（可在设置恢复） |

权重表存储在本地 DB，影响 SignalAggregator 的粗筛逻辑。

### Level 2：Prompt 记忆

用户的偏好/反馈定期总结为自然语言，注入 Agent 的 system prompt：

```
用户偏好摘要（自动生成）：
- 用户对老板发的消息非常敏感，优先级应提高
- 用户不关心营销类推送和系统更新通知
- 用户通常周五下午规划下周
- 用户关注项目 X 的所有动态
```

每周自动压缩一次，防止 prompt 无限增长。

### 条目交互

| 操作 | 触发方式 | 效果 |
|------|---------|------|
| 展开详情 | 点击条目 | 显示原始信号摘要 + Agent 的 reason + suggestion |

{item.source_content}

## 十一、数据模型（概要）

### Signal 表

```
Signal {
    id: Long (PK)
    source_type: String  // notification / calendar / feishu_msg / feishu_doc / chat / time
    source_ref: String   // 原始信号的外部 ID
    title: String        // 信号标题/摘要
    content: String      // 信号完整内容
    timestamp: Long      // 信号发生时间
    processed: Boolean   // 是否已被 Agent 处理
    created_at: Long     // 入库时间

```

```
BoardItem {
    id: String (PK, UUID)
    title: String
    source_type: String
    source_ref: String
    source_content: String
    urgency: String      // high / medium / low
    category: String     // action / attention / info
    reason: String
    suggestion: String
    signal_time: Long
    status: String       // active / completed / dismissed
    completed_at: Long?
    board_date: String   // 归属日期 (yyyy-MM-dd)

```
UserPreference {
    id: Long (PK)
    source_type: String?
    keyword: String?

```

```
FocusRule {
    id: Long (PK)
    content: String      // 自然语言规则

}

## 十二、生命周期

### 一天的流程

```
00:00  清理：昨日已完成条目归档，重置看板
08:00  定时触发：Agent 生成今日看板 v1
       (基于昨晚到现在的信号 + 今日日历)
09:15  新信号：收到飞书消息
10:32  新信号：文档变更
10:45  增量触发：累积 5 条新信号，Agent 更新看板
12:00  定时触发：Agent 刷新看板（午间版）
14:00  用户打开看板，查看、完成 2 条 TODO
15:30  新信号累积，增量触发更新
18:00  定时触发：Agent 生成晚间总结版
23:59  归档今日数据，准备明日
```

### 条目状态流转

```
[active] ──完成──→ [completed] ──次日凌晨──→ [archived]
   │
   └──忽略──→ [dismissed] ──次日凌晨──→ [archived]
```

## 十三、风险与约束

| 风险 | 影响 | 缓解措施 |
|------|------|---------|
| Agent 推理成本 | 一天 5-8 次调用，token 消耗 | 可配置模型、可限制触发频率 |
| 电量消耗 | 后台服务 + 网络请求 | 智能策略、仅 WiFi 模式 |
| 隐私敏感 | 读取所有通知内容 | 本地处理为主、明确权限说明、数据不上传 |
| Agent 幻觉 | 生成不存在的 TODO | 所有条目必须溯源到具体信号 |
| 信息过载 | 看板条目太多反而没用 | 硬限制：每次最多 10 条 action + 5 条 attention |
| NotificationListenerService 被杀 | 丢失通知 | 系统会自动重启；前台补偿兜底 |

## 十五、已决策项（原开放问题）

| # | 问题 | 决策 |
|---|------|------|
| 1 | 看板条目上限 | 默认 action ≤ 8, attention ≤ 5, info ≤ 3（合计约 16 条一屏可览）。设置中提供"看板密度"三档：精简/标准/详尽 |
| 2 | 通知过滤粒度 | MVP 做"按 app 开关 + 隐式学习"。关键词过滤后续按需加 |
